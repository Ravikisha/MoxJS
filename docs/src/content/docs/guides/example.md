The `examples/basic` directory is a minimal, fully working MFJS workspace. It wires together a **shell** host and a **dashboard** remote using Rspack Module Federation, the `@mfjs/runtime` routing library, and the `mfjs` CLI.

---

## Workspace structure

```
examples/basic/
  apps/
    shell/                 # Host app — port 3000
      src/
        bootstrap.tsx      # React entry: router, NavLink, RemoteOutlet
        index.ts           # Async boundary (imports bootstrap.tsx)
        remotes.d.ts       # TypeScript ambient type for 'dashboard/App'
      mfjs.app.json
      mfjs.federation.json
      rspack.config.mjs

    dashboard/             # Remote app — port 3001
      src/
        remote.tsx         # Exposed as './App' — uses RemoteApp
        mfjs.routes.ts     # Auto-generated by `mfjs routes`
        pages/
          index.tsx        # Route: /
          settings.tsx     # Route: /settings
          users/
            [id].tsx       # Route: /users/:id
      mfjs.app.json
      mfjs.federation.json
      rspack.config.mjs
```

---

## Running the example

```sh
cd examples/basic
pnpm install
pnpm dev
```

This starts both apps concurrently:

- Shell host: `http://localhost:3000`
- Dashboard remote: `http://localhost:3001`

Open `http://localhost:3000` in a browser. The shell loads the dashboard remote inside its `<main>` area.

## How the host works (`shell/src/bootstrap.tsx`)

The shell is a small React app. Its three responsibilities are:

1. **Initialise the router** at module level.
2. **Render navigation links** using `NavLink`.
3. **Mount the matching remote** using `RemoteOutlet`.

```tsx
import React from 'react';
import ReactDOM from 'react-dom/client';
import { NavLink, RemoteOutlet, usePathname, getRouter } from '@mfjs/runtime';
import type { RouteTarget } from '@mfjs/runtime';

const HOST_ROUTES: RouteTarget[] = [
  { path: '/dashboard/*', remote: 'dashboard', module: './App' },
  { path: '/',            remote: 'dashboard', module: './App' },
];

const REMOTES = {
  dashboard: () => import('dashboard/App'),
};

// Call at module level so the singleton survives StrictMode double-invocation
getRouter();

function App() {
  const pathname = usePathname();
  return (
    <div>
      <header data-testid="shell-header">
        <NavLink to="/" label="Home" />
        <NavLink to="/dashboard/settings" label="Settings" />
        <NavLink to="/dashboard/users/42" label="User 42" />
        <span data-testid="current-path">{pathname}</span>
      </header>
      <main>
        <RemoteOutlet routes={HOST_ROUTES} remotes={REMOTES} />
      </main>
    </div>
  );
}

ReactDOM.createRoot(document.getElementById('root')!).render(
  <React.StrictMode><App /></React.StrictMode>
);
```

### Route table

`HOST_ROUTES` maps URL prefixes to remote apps:

| Pattern | Remote | Module |
|---|---|---|
| `/dashboard/*` | `dashboard` | `./App` |
| `/` | `dashboard` | `./App` |

Both patterns load the same `dashboard/App` module. Inside the remote, `RemoteApp` handles sub-path resolution.

### Why `getRouter()` at module level?

React StrictMode invokes effects twice in development. If the router were created inside a `useEffect`, the cleanup would call `router.destroy()`, removing the `popstate` listener before the app finishes mounting. Calling `getRouter()` at module level creates the singleton once and ensures the listener is never destroyed.

### Native federation imports

`REMOTES.dashboard` is `() => import('dashboard/App')`. This is a **native Rspack federation import**, not a runtime `loadRemoteModule()` call. It allows Rspack's `ModuleFederationPlugin` to bridge the React share scope automatically, preventing React from loading twice and avoiding `Invalid hook call` errors.

### TypeScript ambient type

`src/remotes.d.ts` declares the module so TypeScript accepts the import:

```ts
declare module 'dashboard/App' {
  import type React from 'react';
  const RemoteApp: React.ComponentType<{ subpath?: string }>;
  export default RemoteApp;
}
```

## How the remote works (`dashboard/src/remote.tsx`)

The remote exposes a single component (`./App`) that delegates page rendering to `RemoteApp`:

```tsx
import { RemoteApp } from '@mfjs/runtime';
import { pages } from './mfjs.routes.js';

export default function RemoteRoot({ subpath = '/' }: { subpath?: string }) {
  return <RemoteApp subpath={subpath} pages={pages} />;
}
```

`RemoteApp` matches `subpath` against the `pages` array, lazy-loads the matching page module, and renders it inside a `<div data-testid="remote-loaded">`.

---

## File-based routing (`mfjs.routes.ts`)

The `pages` array is auto-generated by `mfjs routes`. You can regenerate it any time:

```sh
cd examples/basic/apps/dashboard
mfjs routes
```

Current output:

```ts
import type { RemotePageRoute } from '@mfjs/runtime';
export const pages: RemotePageRoute[] = [
  { path: '/users/:id', load: () => import('./pages/users/[id].tsx') },
  { path: '/settings',  load: () => import('./pages/settings.tsx') },
  { path: '/',          load: () => import('./pages/index.tsx') },
];
```

## Page components

### `pages/index.tsx`

Renders with `data-testid="page-home"`. Includes buttons that navigate to settings and to a user page:

```tsx
import { dispatchMfjsNavigate } from '@mfjs/runtime';

export default function HomePage() {
  return (
    <div data-testid="page-home">
      <h1>Home</h1>
      <button
        data-testid="nav-to-settings"
        onClick={() => dispatchMfjsNavigate({ to: '/dashboard/settings' })}
      >
        Go to Settings
      </button>
      <button
        data-testid="nav-to-user"
        onClick={() => dispatchMfjsNavigate({ to: '/dashboard/users/42' })}
      >
        Go to User 42
      </button>
    </div>
  );
}
```

### `pages/settings.tsx`

Renders with `data-testid="page-settings"` and a back button:

```tsx
export default function SettingsPage() {
  return (
    <div data-testid="page-settings">
      <h1>Settings</h1>
      <button
        data-testid="back-home"
        onClick={() => dispatchMfjsNavigate({ to: '/' })}
      >
        Back to Home
      </button>
    </div>
  );
}
```

### `pages/users/[id].tsx`

Receives a `params` prop from `RemoteApp` with the matched URL parameters:

```tsx
export default function UserPage({ params }: { params: Record<string, string> }) {
  return (
    <div data-testid="page-user">
      <h1>User</h1>
      <span data-testid="user-id">{params.id}</span>
    </div>
  );
}
```

When the URL is `/dashboard/users/42`, `params.id` is `'42'`.

## End-to-end tests

The `tests/e2e/basic-example.spec.ts` Playwright suite covers all 11 scenarios:

| Test | What it checks |
|---|---|
| Loads dashboard on `/` | `page-home` is visible |
| NavLink to settings | `page-settings` is visible after clicking the Settings link |
| NavLink to user | `page-user` is visible, `user-id` shows `42` |
| In-page nav button to settings | Button inside remote navigates to settings |
| In-page nav button to user | Button inside remote navigates to user page |
| Back button from settings | Returns to `page-home` |
| Browser back from settings | Popstate works correctly |
| Browser back from user | Popstate works correctly |
| Current path indicator | `current-path` span updates on every navigation |
| NavLink active style | Active link receives `activeStyle` |
| Params extracted correctly | `user-id` reflects URL segment |

Run the suite:

```sh
MFJS_E2E=1 pnpm e2e        # opt-in local run
pnpm e2e:ci                 # always-on CI run
```
